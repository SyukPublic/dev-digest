# server — INSIGHTS

## What Doesn't Work
- [2026-06-19] A failed `JobRunner` job rethrows into the returned `done` promise, but callers fire-and-forget `enqueue()`, so the rejection goes unhandled and crashes the whole API; guard with `void done.catch(() => {})` plus a global `unhandledRejection` net; `server/src/platform/jobs.ts` (enqueue).

## Codebase Patterns
- [2026-06-22] The `repos` and `pull_requests`/`pr_files`/`pr_commits` tables are cross-module (read/written by repos, pulls, polling, workspace), so their repositories are promoted to the container (`container.reposRepo` / `container.pullsRepo`) like `agentsRepo`/`reviewRepo` — feature code reaches them via the container, never by importing another module's `repository.ts`; row types come from `db/rows.ts`; `server/src/platform/container.ts`.
- [2026-06-22] The PR-list rollups (latest score, total cost, finding severity tally) query the REVIEWS domain tables (`reviews`/`findings`/`agent_runs`), so they live on `ReviewRepository` (`latestReviewScores`/`runCostRows`/`findingSeverityRows`) and the pulls list reads them through `container.reviewRepo` — they are NOT pulls queries; grouping stays pure (`pulls/helpers.ts latestScoreByPr`, `cost.ts`, `findings-summary.ts`).
- [2026-06-22] `GET`/`PUT /settings` intentionally have NO Zod response schema: the `Settings` contract is `SettingsKnown.passthrough()` WITH per-key `.default(...)`, so serializing a response through it would inject defaults the workspace never set (a behavior change). Endpoints whose contracts have no defaults (secrets-status, test-connection) do get a response schema; `server/src/modules/settings/routes.ts`.
- [2026-06-19] `@devdigest/shared` is TWO independent vendored copies (`server/src/vendor/shared` + `client/src/vendor/shared`) with NO sync script — add/change a contract field in BOTH or the other package never sees it; `server/src/vendor/shared/contracts`. [CORRECTION 2026-06-22] A sync script now exists: the SERVER copy is the source of truth (reviewer-core aliases to it too); run `node scripts/sync-shared.mjs` after editing a contract and CI fails on drift via `--check` in `client.yml`. Don't hand-edit the client copy.
- [2026-06-19] The review engine already returns `costUsd` (OpenRouter `usage.cost`, else injected `estimateCost`/`PriceBook`) — persist it, never recompute; surfacing run cost costs zero extra model calls; `server/src/modules/reviews/run-executor.ts` (runOneAgent).
- [2026-06-20] PR-list COST = sum of ALL the PR's agent runs (every batch), skipping unpriced (failed/cancelled) ones; absent when no priced run → list shows "—"; `server/src/modules/pulls/cost.ts` (totalCostByPr). (Was latest-batch-only via `batch_id` until 2026-06-20; `batch_id` is still stamped per `runReview` fan-out but no longer drives the list rollup.)
- [2026-06-20] The PR-list route `GET /repos/:id/pulls` declares only `schema: { params }` — NO response schema; it returns `Promise<PrMeta[]>` as a TS type, so `PrMeta` is type-only there and serialization does NOT strip unknown keys. Surfacing a new list field = add it to the object built in the handler's `rows.map(...)` AND to the `PrMeta` Zod contract (the latter only for types); `server/src/modules/pulls/routes.ts`. Per-PR rollups follow one pattern: an `inArray(prIds)` query + a pure JS grouper (`cost.ts` totalCostByPr, `findings-summary.ts` findingCountsByPr).
- DB-backed tests must use the `*.it.test.ts` suffix or the unit/integration split breaks.
- Secrets are read only through `LocalSecretsProvider` (`~/.devdigest/secrets.json`, 0600);
  `GITHUB_TOKEN` is canonical, `GITHUB_PAT` is an accepted fallback.
- The engine reaps orphaned `running` runs on boot.

- [2026-06-23] Skills was a SCAFFOLD-COMPLETION, not a from-scratch feature: the AGENT side already existed (`agent_skills` table + `AgentsRepository.linkedSkills/setSkills/linkSkill` + `POST/GET /agents/:id/skills` + reviewer-core `assemblePrompt`'s `## Skills / rules` slot + `prompt_assembly.skills` in the trace + the client trace render). What was missing: the skills CRUD module (`modules/skills`), the import adapter, the UI, and — the key wiring — `run-executor.runOneAgent` did NOT load linked skills. Wiring = load `this.agents.linkedSkills(agent.id)`, filter to ENABLED, map to `SkillInput[]` and pass as `skills` to `reviewPullRequest` (omit when empty so the prompt stays byte-identical to the no-skills baseline). `server/src/modules/reviews/run-executor.ts`.
- [2026-06-23] Skill TRUST is decided server-side from `skills.source` and passed INTO the pure engine: `reviewPullRequest`/`assemblePrompt` take `skills?: SkillInput[]` where `SkillInput = string | { body; trusted? }`. Trusted = `source ∈ {manual, extracted}` → body rendered verbatim as instructions; untrusted (imported) → `assemblePrompt` wraps it with `wrapUntrusted('skill-i', …)` so the injection guard treats it as DATA. reviewer-core stays pure (wrapping is text-only); that wrap is the ONLY core change. Imports (file/URL/zip) all store `source='imported_url'` (the generic "Imported" bucket — `SkillSource` has no per-channel value) and save DISABLED-until-vetted.
- [2026-06-23] Per-prompt-section token counts surface in the trace via `prompt_assembly.tokens` (a NEW optional `Record<string,number>` on the `PromptAssembly` contract). The pure engine leaves it unset (no tokenizer in reviewer-core); the server fills it in `run-executor.sectionTokens()` using `container.tokenizer` when persisting the trace. Optional so old traces still parse; `server/src/vendor/shared/contracts/trace.ts`.
- [2026-06-23] A skill is versioned ONLY on a body change (`skill_versions`, mirrors `agent_versions`) — editing name/description/type or toggling enabled does NOT bump the version (`isBodyChange` in `modules/skills/helpers.ts`).
- [2026-06-23] Conventions Extractor (`conventions-extract` job) is BEST-EFFORT by design: deterministic config-file rules (`config-extractor.ts`, confidence 1.0, no LLM) are ALWAYS persisted; the LLM pass is additive, wrapped in try/catch AND bounded by `EXTRACTION_TIMEOUT_MS` (< JobRunner's 120s), so a slow/keyless model can't fail the whole scan. Every LLM candidate is verified against disk (snippet anchor must literally exist in the cited sample) before persist → drops hallucinated `file:line`; `server/src/modules/conventions/service.ts` (runExtractJob), `helpers.ts` (verifyAndCorroborate). The `conventions` table pre-existed empty (course scaffold) — only ADD columns. GOTCHA: config rules read only the clone ROOT (`tsconfig.json`, `.prettierrc`, …); a monorepo with per-package configs yields zero config rules.
- [2026-06-24] `replaceAll` carries `accepted=true` across re-scans by normalised rule key: it SELECTs prior accepted rows INSIDE the same transaction before delete+insert, so no concurrent scan sees an empty-table window and no race can mix states. The preserve logic lives in the repository (not the service) specifically to keep the read inside the tx — moving it to the service would reopen the race; `server/src/modules/conventions/repository.ts` (replaceAll), `helpers.ts` (acceptedRuleKeys).
- [2026-06-24] `RunLogger.step()` re-throws on failure — do NOT wrap best-effort / swallowed blocks in it; use manual `runLog.tool()` + `runLog.info()` in catch instead (e.g. the LLM pass in `runExtractJob`); `server/src/platform/run-logger.ts` (step), `server/src/modules/conventions/service.ts` (runExtractJob).
- [2026-06-24] `runBus.complete(jobId)` MUST be in a `finally {}` block in every job body — if omitted the client `EventSource` hangs until `onerror` fires; success, early return, and thrown errors are all exit paths that must close the stream; `server/src/platform/sse.ts` (RunBus), `server/src/modules/conventions/service.ts` (runExtractJob).
- [2026-06-24] `RunBus.subscribe()` replays its event buffer on connect and `onDone()` fires immediately for already-completed runs — a late SSE subscriber (EventSource opened after a fast job finishes) still receives all events and the done signal; no special "already finished?" handling needed; `server/src/platform/sse.ts` (RunBus, streamRunEvents).
- [2026-06-24] Conventions corroboration/dedup enrichment (F3/F4) reuses EXISTING container ports — no new adapters. F3: `container.astGrep` gains a SECOND consumer beyond the indexer (structural corroboration); the service parses each sample ONCE into a `Map<path, ParsedSymbol[]>` via the pure `buildSymbolIndex(astGrep, contents)` and passes it to `verifyAndCorroborate(…, parsedIndex?)` — parse-once-in-service, not per-candidate. F4: `container.embedder()` (gated by `EMBEDDINGS_ENABLED`, THROWS `ConfigError` when off) wired into `semanticDedup(drafts, vectors, threshold)`. Both are best-effort: NOT in `runLog.step()` (re-throw, see above), wrapped in try/catch, degrading to the text/string-dedup result; the embedder call is also skipped unless `drafts.length > 1`. Helpers stay pure (port/vectors passed as args). `SEMANTIC_DEDUP_THRESHOLD=0.92` is a tunable start; `server/src/modules/conventions/{service,helpers,constants}.ts`.

## Tool & Library Notes
- [2026-06-22] Onion boundaries are enforced by `pnpm arch:check` (dependency-cruiser, `server/.dependency-cruiser.cjs`; inlined in `server-unit.yml` typecheck job). `no-circular` is `warn` NOT `error` on purpose — the hand-rolled DI passes the whole `Container` into services while the container constructs `RepoIntelService`, an intentional composition-root cycle. Pure-logic helpers now live in `server/src/lib/` (`extract.ts`, `diff-parser.ts`), NOT under `adapters/`. `no-concrete-adapters-in-services` is now exception-free: astgrep got a container port (`AstGrep` interface + `AstGrepAdapter` in `adapters/astgrep/index.js`, `container.astGrep`), so repo-intel service/pipeline reach the parser via the container, not a direct import. GOTCHA: unit tests that build a FAKE container object (e.g. `indexer-pipeline.test.ts` `makeContainer`) must now provide `astGrep: new AstGrepAdapter()` or the pipeline throws `Cannot read properties of undefined (reading 'langForFile')`.
- [2026-06-19] pnpm ≥10 blocks dependency build scripts by default; pnpm 11 replaced `onlyBuiltDependencies` with an `allowBuilds` map (`name: true|false`) — run `pnpm approve-builds` (needed for esbuild/ssh2/cpu-features/protobufjs); `server/pnpm-workspace.yaml`.
- [2026-06-23] `fflate` (pure JS, zero post-install scripts → NO `pnpm approve-builds` needed) powers zip skill imports. The import adapter `adapters/skill-import` decompresses ONLY markdown via `unzipSync(bytes, { filter })` (executable entries never enter memory) and SSRF-guards URL fetch (http(s) only; resolved IP must be public via `dns.lookup`; redirects rejected). It's a container port (`container.skillImporter` + `ContainerOverrides.skillImporter`) so tests inject a fake — no network/fs. GOTCHA: install `fflate` from the WSL toolchain (`Ubuntu-24.04-dev-digest-test`), NOT Git Bash pnpm — Git Bash pnpm tries to PURGE the WSL-built `node_modules`.

## Recurring Errors & Fixes
- Migrations are NOT applied on boot — first-run `relation ... does not exist` = run `pnpm db:migrate`.
- [2026-06-19] On a repo path containing spaces, the `file://${process.argv[1]}` entrypoint guard never equals `import.meta.url` (which percent-encodes spaces), so `pnpm db:migrate`/`db:seed` exit 0 doing nothing; use `pathToFileURL(process.argv[1]).href`; `server/src/db/migrate.ts` + `seed.ts`.
- [2026-06-24] Unit test mocks for services that use `RunLogger` must include `runBus: { publish: vi.fn(), complete: vi.fn() }` in the fake container — otherwise the `finally { runBus.complete() }` throws `Cannot read properties of undefined`; mirrors the existing `astGrep` container mock requirement; `server/test/conventions-service.test.ts`.
