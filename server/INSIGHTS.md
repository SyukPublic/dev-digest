## What Doesn't Work
- [2026-06-19] A failed `JobRunner` job rethrows into the returned `done` promise, but callers fire-and-forget `enqueue()`, so the rejection goes unhandled and crashes the whole API; guard with `void done.catch(() => {})` plus a global `unhandledRejection` net; `server/src/platform/jobs.ts` (enqueue).
- [2026-06-27] Smart-diff overlays the latest review's findings onto the CURRENT diff by line NUMBER with NO SHA/content check, so after later commits a deleted-file finding silently vanishes from the overlay and a moved/changed-line finding mis-anchors onto unrelated code (the GitHub-comment `is_outdated` path is a SEPARATE, human-only mechanism). Per-finding actuality is just the GROUNDING predicate re-applied to the CURRENT diff — `lib/diff-parser.ts` already exposes `hunk.newLineNumbers` and `diffFromPrFiles` rebuilds the current `UnifiedDiff` with no network — see `docs/specs/review-freshness.md` Stage 2; `server/src/modules/smart-diff/service.ts`.
- [2026-06-27] Isolated `*.it.test.ts` runs are COLD (JIT, first pool connection, Docker page cache un-warmed), so a fire-and-forget mock review's ~15 sequential DB round-trips can take 10-18s — vs <2s once the full suite has warmed them; the flake is INVERTED (isolated `pnpm exec vitest run test/reviews.it.test.ts` fails ~1/4, the warm full suite hides it). A poll-waiter with a tight budget that silently RETURNS its non-terminal rows on timeout then makes the caller read `[]` from `/reviews` and crash with a cryptic `Cannot read properties of undefined (reading 'findings')` — such a waiter must instead THROW on timeout (naming the run statuses it saw) and budget generously (30s); `server/test/helpers/runs.ts` (waitForPrRuns).
- [2026-06-27] L1 `anchorStatus` can NEVER report `moved_out`/`orphaned` for a finding on a NEWLY-ADDED file: an added file is one whole-file hunk (`@@ -0,0 +1,N @@`) so every line 1..N is "covered" → the range always intersects → `current`, even when the code on those lines was rewritten by a later commit (content change is invisible — that is L2, not L1). To exercise `moved_out`/`orphaned` you need a MODIFY/DELETE diff, NOT an added-file PR; `reviewer-core/src/grounding.ts` (anchorStatus), `server/src/modules/reviews/service.ts` (reviewsForPull).
- [2026-06-28] `content_changed` read-path footgun: `anchorStatus(f) === 'current'` does NOT imply `anchoredText(f)` is non-null — FULL-FILE kinds (`secret_leak`/`lethal_trifecta`/`phantom`/`hook`) anchor on file-presence, not line range, so their `start..end` can fall outside every current hunk → `anchoredText`→null while status is `current`. The fingerprint compare MUST guard `cur != null` (`cur != null && cur !== stored`); otherwise an uncomputable current fp false-flags `content_changed` against a non-null stored fp (write/read hash symmetry alone doesn't save you — the stored fp was non-null when those lines WERE in a hunk at review time); `server/src/modules/reviews/service.ts` (reviewsForPull).
- [2026-07-01] Blast "0 callers" on a `status=full` index whose `symbols`/`file_edges` are correct = an UNRESOLVED index: `references.decl_file` is all-NULL so `getResolvedCallers` (filters `decl_file IN changedFiles`) returns empty for EVERY change. Root cause is SYSTEMIC, not a stale snapshot: a full index of a real repo takes ~165-198s > JobRunner's 120s `withTimeout`, so the `repo-intel-index` job is marked `failed` (`TimeoutError`, deliberately NOT retryable) while `runFullIndex` keeps running UNCANCELLED as a zombie (`withTimeout` races a timer, it can't cancel the promise); `INDEX_SOFT_BUDGET_MS≈110s` doesn't save it because it only gates the enqueue loop, not the parse-workers + dependency-cruiser graph build, so `softBudgetReached` stays false and status is still stamped `full`. With polling (5-min) + refresh re-enqueuing and PQueue concurrency=3 + NO per-repo dedup, overlapping zombies race on the destructive `deleteAllForRepo`→insert→`resolveReferences` sequence → refs present but decl_file unresolved. Diagnose: `count(*) FILTER (WHERE decl_file IS NOT NULL)` (0/N = unresolved) + the `jobs` table (index kind: many `failed` "timed out after 120000ms" vs ~1 `done`); durable fix needs a per-kind longer index timeout, a per-repo concurrency guard, and a transactional delete→insert→resolve swap; `server/src/platform/{jobs.ts,resilience.ts}` (withTimeout), `server/src/modules/repo-intel/pipeline/full.ts` (runFullIndex), `repository.ts` (getResolvedCallers, deleteAllForRepo).
- [2026-07-01] `RepoIntelRepository.resolveReferences({reset:true})` (incremental) was two NON-atomic `db.execute`s — `UPDATE … SET decl_file=NULL` then the resolve UPDATE — so a crash/kill between them persists all-NULL; now wrapped in one `db.transaction` (reset visible only WITH the re-resolve). Defense-in-depth against the unresolved-index footgun above: `tryPersistentBlast` treats `total>0 && resolved===0` as a defective index (via `getReferenceResolution`) and returns null → `getBlastRadius` falls back to the ripgrep best-effort instead of asserting a confident `degraded:false` "no callers"; probe only fires when callers are empty; `server/src/modules/repo-intel/{repository.ts,service.ts}` (resolveReferences, getReferenceResolution, tryPersistentBlast).
- [2026-07-01] JobRunner marks a handler `failed` when it exceeds `timeoutMs` (default 120s), but `withTimeout` only rejects the WRAPPER — the handler promise keeps running to completion (JS can't cancel a promise) and still performs its DB writes (e.g. stamps `repo_index_state` with `dur_ms` > the cap), so `jobs.status='failed'` does NOT mean the work didn't happen (a heavy clone/index can "fail" yet persist a full result minutes later). There is also NO per-repo concurrency guard, so polling/refresh re-enqueues + PQueue concurrency(3) let two zombie runs of the SAME repo overlap and race on read-modify-write — any heavy delete-then-insert handler must assume a concurrent peer; `server/src/platform/{jobs.ts,resilience.ts}` (enqueue, withTimeout).
- [2026-07-01] Fix (a) raised the index pipeline's per-kind JobRunner hard timeout to `INDEX_JOB_TIMEOUT_MS` (600s) but deliberately LEFT `INDEX_SOFT_BUDGET_MS` at 110s. Do NOT raise the soft budget to "track" the longer hard cap: it gates only the fast enqueue loop, while the EXPENSIVE phase (`parseQ.onIdle` + dependency-cruiser graph) runs AFTER it — bumping soft to ~hard lets the enqueue phase eat the budget and starves the graph, re-creating the hard-cap kill/zombie. To make the soft budget meaningful it must be RE-SCOPED to also gate parse+graph, not just raised; `server/src/modules/repo-intel/{constants.ts,pipeline/full.ts}` (INDEX_SOFT_BUDGET_MS, runFullIndex).
- [2026-07-02] A blast PR-vs-index freshness signal must NOT be `state.lastIndexedSha !== pull.headSha` (nor `prBranch !== indexedBranch`): the index is built on `origin/<defaultBranch>` so a PR head is ALWAYS a different commit → the flag is permanently true → noise → the confident-wrong "0 downstream" stays invisible (worse than no badge). The meaningful, NON-constant signal (TD-003, checked empty_map-first): `empty_map` = a readable-but-empty downstream (always caveat it — the dangerous case) OR `base_diverged` = `pull.base !== indexedBranch` (a PR NOT targeting the default branch); `server/src/modules/blast/freshness.ts` (deriveBlastFreshness), `service.ts` (getBlast).
- [2026-07-02] The FULL server suite (`pnpm test`) can fail ONE `*.it.test.ts` with `Error: Failed to connect to Reaper` (testcontainers Ryuk) at `startPg()` while other Testcontainers-pg tests in the SAME run pass — a race when many pg containers spin up in parallel, NOT a code regression (distinct from the cold-start flake above); it passes on isolated re-run (`pnpm exec vitest run <file>`). Don't chase it as a real failure; `server/test/helpers/pg.ts` (startPg).

## Codebase Patterns
- [2026-06-22] The `repos` and `pull_requests`/`pr_files`/`pr_commits` tables are cross-module (read/written by repos, pulls, polling, workspace), so their repositories are promoted to the container (`container.reposRepo` / `container.pullsRepo`) like `agentsRepo`/`reviewRepo` — feature code reaches them via the container, never by importing another module's `repository.ts`; row types come from `db/rows.ts`; `server/src/platform/container.ts`.
- [2026-06-22] The PR-list rollups (latest score, total cost, finding severity tally) query the REVIEWS domain tables (`reviews`/`findings`/`agent_runs`), so they live on `ReviewRepository` (`latestReviewScores`/`runCostRows`/`findingSeverityRows`) and the pulls list reads them through `container.reviewRepo` — they are NOT pulls queries; grouping stays pure (`pulls/helpers.ts latestScoreByPr`, `cost.ts`, `findings-summary.ts`).
- [2026-06-22] `GET`/`PUT /settings` intentionally have NO Zod response schema: the `Settings` contract is `SettingsKnown.passthrough()` WITH per-key `.default(...)`, so serializing a response through it would inject defaults the workspace never set (a behavior change). Endpoints whose contracts have no defaults (secrets-status, test-connection) do get a response schema; `server/src/modules/settings/routes.ts`.
- [2026-06-19] `@devdigest/shared` is TWO independent vendored copies (`server/src/vendor/shared` + `client/src/vendor/shared`) with NO sync script — add/change a contract field in BOTH or the other package never sees it; `server/src/vendor/shared/contracts`. [CORRECTION 2026-06-22] A sync script now exists: the SERVER copy is the source of truth (reviewer-core aliases to it too); run `node scripts/sync-shared.mjs` after editing a contract and CI fails on drift via `--check` in `client.yml`. Don't hand-edit the client copy.
- [2026-06-19] The review engine already returns `costUsd` (OpenRouter `usage.cost`, else injected `estimateCost`/`PriceBook`) — persist it, never recompute; surfacing run cost costs zero extra model calls; `server/src/modules/reviews/run-executor.ts` (runOneAgent).
- [2026-06-20] PR-list COST = sum of ALL the PR's agent runs (every batch), skipping unpriced (failed/cancelled) ones; absent when no priced run → list shows "—"; `server/src/modules/pulls/cost.ts` (totalCostByPr). (Was latest-batch-only via `batch_id` until 2026-06-20; `batch_id` is still stamped per `runReview` fan-out but no longer drives the list rollup.)
- [2026-06-20] The PR-list route `GET /repos/:id/pulls` declares only `schema: { params }` — NO response schema; it returns `Promise<PrMeta[]>` as a TS type, so `PrMeta` is type-only there and serialization does NOT strip unknown keys. Surfacing a new list field = add it to the object built in the handler's `rows.map(...)` AND to the `PrMeta` Zod contract (the latter only for types); `server/src/modules/pulls/routes.ts`. Per-PR rollups follow one pattern: an `inArray(prIds)` query + a pure JS grouper (`cost.ts` totalCostByPr, `findings-summary.ts` findingCountsByPr).
- DB-backed tests must use the `*.it.test.ts` suffix or the unit/integration split breaks.
- Secrets are read only through `LocalSecretsProvider` (`~/.devdigest/secrets.json`, 0600);
  `GITHUB_TOKEN` is canonical, `GITHUB_PAT` is an accepted fallback.
- The engine reaps orphaned `running` runs on boot.

- [2026-06-23] Conventions Extractor (`conventions-extract` job) is BEST-EFFORT by design: deterministic config-file rules (`config-extractor.ts`, confidence 1.0, no LLM) are ALWAYS persisted; the LLM pass is additive, wrapped in try/catch AND bounded by `EXTRACTION_TIMEOUT_MS` (< JobRunner's 120s), so a slow/keyless model can't fail the whole scan.
