## Orchestration Patterns
- [2026-06-25] The biggest hidden cost in PARALLEL subagent runs is each implementer/verifier independently re-reading the same template/convention files (template agents, contracts, shared boilerplate). Fix: `implementation-planner` emits a **Shared scaffold (context pack)** section that lifts reusable boilerplate VERBATIM with `file:line`, phases REFERENCE it instead of "go read there", and the orchestrator embeds that pack into each implementer's spawn prompt â€” pass ready fragments, not pointers. Codified in `.claude/agents/implementation-planner.md` (Working loop "Context-pack rule" + spec template) and `.claude/agents/implementer.md` (Working loop step 1).
- [2026-06-25] Surface skills load ON-DEMAND via the `Skill` tool, not preloaded; only truly always-on skills go in the `skills:` frontmatter (read-only/coverage-focused agents like `plan-verifier` omit the field entirely). Keeps each parallel agent's context lean â€” it loads only the set for the surface its slice touches; `.claude/agents/implementer.md` / `implementation-planner.md` (frontmatter `skills:` comment).
- [2026-06-25] `effort` is an official PER-SUBAGENT frontmatter field (values `low`/`medium`/`high`/`xhigh`/`max`; overrides the session effort, default = inherit; available levels are model-dependent â€” `xhigh`/`max` need opus/fable, sonnet caps lower). Tune by ROLE: `xhigh` on reasoning-heavy opus agents (`architecture-reviewer`, `plan-verifier`, `implementation-planner`); keep the parallel `implementer` at `high` so a fan-out of opus workers can't inherit an `xhigh`/`ultracode` session and multiply cost. Gotcha: settings.json `effortLevel` documents only `low`..`xhigh` (no `max`), but frontmatter `effort` includes `max`. Verified vs code.claude.com sub-agents + settings docs 2026-06-25.

## What Doesn't Work
- [2026-06-25] A `PreToolUse` hook in `.claude/settings.json` does NOT fire for subagents â€” each subagent runs its own session with its own hooks. A hard per-path write boundary for a subagent is only enforceable via a `hooks:` block in that agent's OWN frontmatter; a `tools:` whitelist limits WHICH tools exist (e.g. no `Write`/`Edit`) but never WHICH paths. Verified against Claude Code docs 2026-06-25 (hooks.md + subagents.md).
- [2026-06-25] Headless `claude -p` auto-routing does NOT fire for the `test-writer` agent: the main agent treats "write/add/cover tests" as a core inline capability and does it itself rather than delegating (two probes with distinct phrasings both returned no `subagent_type`). REVIEW/produce-artifact agents auto-route fine â€” `architecture-reviewer`, `plan-verifier`, `doc-writer` all routed correctly. So a "does the trigger phrase auto-route?" probe is a valid acceptance check only for delegation-worthy agents; for test-writing, rely on EXPLICIT invocation (name the subagent) and treat agent-type registration (description visible in session) as the validity signal instead. `.claude/agents/test-writer.md`.
- [2026-06-25] The `claude-code-guide` subagent confidently FABRICATES specifics about Claude Code config: asked about per-subagent `effort`, it got the field name right but dropped `max` from the value list and invented a non-existent `permissionMode: read-only`.
