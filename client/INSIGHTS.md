## Codebase Patterns
- [2026-06-22] `useRunEvents` (`lib/hooks/reviews.ts`) intentionally depends on `const key = runIds.join(",")`, NOT the `runIds` array — its `eslint-disable react-hooks/exhaustive-deps` is deliberate. This makes the SSE subscription content-addressed: callers may pass a fresh array each render (same ids ⇒ no re-subscribe), so do NOT "fix" the disable or memoize `runIds` at call sites.
- [2026-06-19] Run-cost UI shares `src/lib/format.ts` (`formatCost` → "—" for unknown cost, never "$0.00") and the `RunCostBadge` component (`compact` | `withTokens`) — reuse them, don't re-format cost inline; `client/src/components/run-cost-badge`.
- Live run logs use a native `EventSource` per active run (`useRunEvents`); on reload the
  log is rebuilt from the persisted trace (`GET /runs/:id/trace`), with a 4s polling fallback.
- Global errors (network/5xx) toast; 4xx is silent → handled inline as empty states.
- Theme is driven by CSS variables + `data-theme`/`data-density`, not a Tailwind theme config.
- [2026-06-20] The PR-list table card (`pulls/styles.ts` `tableCard`) has `overflow:hidden`, so a row-anchored popover positioned `absolute` gets clipped. Render it `position:fixed` from the trigger's `getBoundingClientRect()` and `createPortal` to `document.body`; also `e.stopPropagation()` on the cell + popover so clicks don't trigger the row's navigate-to-PR `onClick`.
- [2026-06-20] One findings popover serves BOTH the PR-list FINDINGS cell and the PR-detail timeline run rows. Because two features consume it, the shared UI was promoted [2026-06-22] out of the PR-list route into the neutral `src/components/findings/` module: `FindingsFilterPopover` (severity chips + list), `FindingPreviewList`, `SeverityCountBadges`, shared `styles.ts`, plus the pure helpers `countBySeverity`/`visibleFindings` (`helpers.ts`) and `SEVERITY_ORDER`/`LOW_CONFIDENCE_THRESHOLD` (`constants.ts`). Import via `@/components/findings/*`. (`KEY_TO_ACTION` stayed in `FindingsPanel/constants.ts` — it's FindingsPanel-only.) Callers resolve `counts` + `findings` and pass an optional `onPick` (list → navigate to PR; timeline → scroll to the run's accordion). The list cell lazy-loads via `usePrReviews` while open; the timeline already has findings in memory (the PR's reviews, keyed by `run_id`).
- [2026-06-20] Severity tally + filter is a reusable primitive `SeverityFilter` (toggle `Chip`s over `SEV` tokens) shared by the popover and each run's `FindingsPanel`; the filter helper is `visibleFindings(findings, hideLow, activeSeverities?)` (now in `@/components/findings/helpers`). A popover resets its filter to "all on" simply by mounting fresh on open (no key hack), since it only renders while open.
- [2026-06-22] Adding an i18n namespace needs NO wiring: `src/i18n/request.ts` (`loadMessages`) `readdirSync`s `messages/<locale>/` and merges every `*.json` as `{ [filename]: {...} }`. Drop `messages/en/<ns>.json` and use `useTranslations("<ns>")` — done. (Test files import a specific `messages/*.json` by relative path; those stay relative since `messages/` lives outside `src/`, so the `@/` alias can't reach them.)
- [2026-06-22] Modal & Drawer (`vendor/ui/kit`) share one a11y hook `useDialogA11y(onClose)` (`kit/useDialogA11y.ts`): Escape-to-close + Tab focus-trap + focus-restore on unmount. Attach the returned ref to the `role="dialog"` node and give it `tabIndex={-1}` + `outline:none`. `onClose` is read through a ref so focus setup runs once on mount (a changing inline-arrow `onClose` won't steal focus mid-interaction). Popovers like `FindingsFilterPopover` keep their own Escape/outside-click — they are not dialogs.

- [2026-06-24] When pushing a conditionally-rendered `Button` to the far end of a flex toolbar, use an unconditional `<div style={{flex:1}} aria-hidden />` spacer rather than `marginLeft:auto` on the button — vendored `Button` doesn't guarantee `style` forwarding (see kit prop-forwarding note below), and the spacer stays correct even when the button is absent; `ConventionsListView/styles.ts` (grow).
- [2026-06-22] kit inputs forward arbitrary DOM props INCONSISTENTLY: `TextInput`/`Textarea` spread `...rest` onto the real `<input>`/`<textarea>` (so `aria-*`, `id`, `data-*` work), but `SelectInput`/`SearchableSelect` do NOT (internal element, no passthrough). `FormField` wires `aria-describedby` to its hint via `cloneElement` on the child — it applies on TextInput/Textarea and is a harmless no-op on the selects. To attach DOM/a11y attrs to a select, extend that component first.
- [2026-06-23] The shared `Markdown` (`vendor/ui/primitives/Markdown.tsx`) styles ONLY inline elements unless you add block renderers: the design reset zeroes h1–h4/p margins and Tailwind preflight flattens heading font-size/weight, so markdown headings/lists render as plain body text until you supply explicit `h1/h2/h3` + `ul/ol/li` component overrides.
- [2026-06-23] To persist a UI reorder/edit across unmount, write the mutation's authoritative response into the cache via `qc.setQueryData(key, data)` — `invalidateQueries` alone loses it, because leaving the page cancels the background refetch and the remounted component re-seeds from STALE cache (even though the DB write succeeded); `lib/hooks/skills.ts` (useSetAgentSkills).
- [2026-06-24] `Modal` (`vendor/ui/kit/Modal.tsx`) body pads via the `bodyPad` prop (default `"20px 24px"`); it previously had ZERO body padding — unlike sibling `Drawer`, whose body is `padding:24` — so naive modals sat flush to the rounded edge while header/footer were inset 24. Modals that manage their own body (already self-padded, or full-bleed like a search bar + scrollable `pre`) MUST pass `bodyPad={0}` to avoid double-padding (`PromptBlock` full prompt view, `CreateAgentModal` whose `styles.ts` `s.body` already pads).
- [2026-06-24] Per-category skill generation (`CreateSkillFromConventionsModal` per-category mode) creates a skill per category by looping `POST /skills` client-side — and `skills.name` has NO unique constraint server-side (`server/src/db/schema/knowledge.ts` skills table), so re-running on the same repo silently creates DUPLICATE skills. This is accepted/out-of-scope by the multi-skill plan (server atomicity + 409 uniqueness deferred), not a bug — don't "fix" it client-side; de-dup belongs in a future server bulk route.
- [2026-06-24] `prevRunningRef` + `useEffect([running])` (single dep) is the correct pattern for detecting SSE stream close (`running: true→false`) without a race condition; using `[running, jobId]` as deps fires the effect prematurely on the render where `jobId` is first set before `useSseEvents` can call `setRunning(true)`; `client/src/lib/hooks/sse.ts` (useSseEvents), `client/src/app/conventions/_components/ConventionsListView/ConventionsListView.tsx`.
- [2026-06-26] The merged Intent block (intent + RISK AREAS in ONE card) has ONE Recompute that fires the two mutations SEQUENTIALLY — `await recomputeIntent.mutateAsync()` THEN `recomputeRisks.mutateAsync()`, NOT `Promise.all`. The server `analyzeRisks` reads the PERSISTED intent (`repo.getIntent`) to anchor risk scope, so parallelizing would recompute risks against the STALE intent. Do not "optimize" to parallel. Combined aria-live derives from both mutations (computing if either pending; `recomputeFailed` if either errors; `briefUpdated` when both succeed); `IntentCard.tsx` (handleRecompute).

## Tooling
- [2026-06-24] Client component tests use `fireEvent` from `@testing-library/react` — `@testing-library/user-event` is NOT a dependency, so `import userEvent` fails typecheck. Pattern: `vi.mock` the data hooks + `next/navigation`, render under `NextIntlClientProvider` (messages imported by relative path, `@/` can't reach `messages/`) + `ToastProvider`; assert toasts via their rendered text. See `AgentEditor.test.tsx`, `FindingCard.test.tsx`, `CreateSkillFromConventionsModal.test.tsx`.
- [2026-06-22] Client ESLint (`eslint.config.mjs`) bridges `eslint-config-next@15` via `FlatCompat` (`@eslint/eslintrc`) because v15 is still eslintrc-style — the native flat export only lands in eslint-config-next@16. Pin `eslint@^9` (config-next@15 isn't built for eslint 10). The TS import resolver pulls native `unrs-resolver`, whose build must be allowed: `pnpm approve-builds` (added to `pnpm-workspace.yaml` `allowBuilds`) — otherwise `pnpm lint` dies in the deps-status pre-check with `ERR_PNPM_IGNORED_BUILDS`. Import boundaries (feature privacy + no deep-relative / `@/app/**`) are enforced by `import/no-restricted-paths` + `no-restricted-imports`; `pnpm lint` runs `--max-warnings 0`.
